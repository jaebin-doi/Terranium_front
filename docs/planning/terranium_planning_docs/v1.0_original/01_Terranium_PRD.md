# Terranium 디지털 트윈 플랫폼 PRD

문서 버전: v1.0  
작성일: 2026-05-08  
제품명: Terranium  
대상: ㈜디오아이 내부 기획, 개발, 영업, 사업 제안, 구축 견적 산정

## 1. 제품 개요

Terranium은 드론 측량 데이터, 3D 디지털 트윈, GeoAI 분석, 재난·안전 시뮬레이션, 보고서 자동화를 하나의 업무 흐름으로 연결하는 B2B SaaS 플랫폼이다.

핵심 목표는 단순한 3D 시각화 도구가 아니라, 현장 데이터 수집부터 데이터 처리, 공간 분석, 위험 예측, 의사결정 보고까지 통합하는 Level 4 디지털 트윈 플랫폼을 제공하는 것이다.

## 2. 배경과 문제 정의

공공기관, 산업단지, 제조 대기업, 도시·국토 관리 조직은 드론 측량 결과물과 공간 데이터를 점점 더 많이 확보하고 있다. 그러나 현업에서는 다음 문제가 반복된다.

- 드론 촬영, 정사영상, LiDAR, Point Cloud, DEM/DSM, 3D Mesh가 각각 다른 도구와 폴더에 흩어져 있다.
- 데이터 처리 결과를 담당자가 직접 다운로드, 변환, 캡처, 보고서화해야 한다.
- 분석은 외부 전문가 또는 개별 툴에 의존하며 재현성과 이력 관리가 어렵다.
- 침수, 지반침하, 균열, 도로 손상 등 위험 분석 결과가 의사결정 시스템과 연결되지 않는다.
- 정부 제출용 보고서, 내부 의사결정 자료, 현장 점검 결과가 매번 수작업으로 만들어진다.
- 데이터 주권, 접근 권한, 감사 로그, 보안 정책을 플랫폼 단위로 관리하기 어렵다.

Terranium은 이 문제를 데이터 파이프라인, 3D Web GIS, GeoAI, 관리자 콘솔, 보고서 생성 기능으로 해결한다.

## 3. 제품 비전

한국의 핵심 공간 데이터를 안전하게 수집하고, 정밀하게 분석하며, 산업과 공공의 의사결정을 가속화하는 Spatial AI 운영 플랫폼.

## 4. 제품 포지셔닝

Terranium은 다음 세 가지 제품군의 교차 영역에 위치한다.

- Web GIS / Digital Twin Viewer
- Drone Mapping Data Platform
- GeoAI Decision Intelligence SaaS

기존 GIS, 3D 뷰어, 드론 데이터 관리 도구와 달리 Terranium은 드론 데이터 수집부터 AI 분석, 시뮬레이션, 보고서, 운영 관리까지 연결하는 End-to-End 업무 플랫폼을 목표로 한다.

## 5. 핵심 사용자

### 5.1 공공기관 관리자

도시 안전, 국토 관리, 재난 대응, 시설물 점검 업무를 담당한다.

주요 니즈:

- 위험 지역을 지도에서 직관적으로 확인
- 분석 결과와 근거 데이터를 함께 검토
- 정부 제출용 보고서 생성
- 권한, 감사 로그, 데이터 반출 이력 관리

### 5.2 산업시설 안전·환경 담당자

공장, 산업단지, 발전소, 교량, 도로, 해양시설 등의 점검과 리스크 관리를 담당한다.

주요 니즈:

- 침수 위험, 균열, 토공량, 지형 변화 분석
- 반복 점검 데이터 비교
- 현장 작업 전 위험 구역 파악
- 내부 보고 및 외부 감사 대응

### 5.3 드론 측량 운영자

현장 촬영, 미션 계획, GCP/CP 검증, 데이터 업로드와 QC를 담당한다.

주요 니즈:

- 프로젝트별 촬영 구역과 비행 경로 관리
- 센서, 고도, 중복도, 목표 GSD 설정
- 현장 QC 체크리스트
- 처리 상태와 오류 확인

### 5.4 GeoAI 분석가

AI 모델 실행, 결과 검수, 위험도 해석, 모델 성능 평가를 담당한다.

주요 니즈:

- 분석 모듈 선택과 실행
- 결과 레이어 비교
- 신뢰도, 정확도, 오류 객체 검수
- 분석 결과를 보고서와 API로 배포

### 5.5 시스템 관리자

조직, 사용자, 권한, API, 인프라, 보안, 백업, 감사 로그를 관리한다.

주요 니즈:

- 역할 기반 접근 제어
- API 키와 사용량 관리
- 모델 배포와 인프라 상태 모니터링
- 보안 정책, 데이터 주권, 백업 복구 관리

## 6. 제품 목표

### 6.1 1차 목표

- 드론 측량 프로젝트를 생성하고 데이터셋을 관리할 수 있다.
- 정사영상, Point Cloud, 3D Mesh, DEM/DSM을 웹에서 시각화할 수 있다.
- 프로젝트별 GeoAI 분석 모듈을 실행하고 결과 레이어를 저장할 수 있다.
- 분석 결과를 보고서 형태로 생성할 수 있다.
- 조직, 사용자, 권한, 감사 로그를 관리할 수 있다.

### 6.2 2차 목표

- 침수, 지반침하, 균열, 도로 손상 등 주요 위험 분석 모델을 운영한다.
- 시간 축 기반 변화 탐지와 비교 분석을 제공한다.
- 시뮬레이션 기반 의사결정 기능을 제공한다.
- API를 통해 외부 시스템과 연동한다.
- 고객별 독립 테넌트와 데이터 주권 정책을 지원한다.

### 6.3 비목표

초기 버전에서 다음은 핵심 범위에서 제외한다.

- 완전 자동 드론 관제 시스템
- 실시간 드론 영상 스트리밍 관제
- 범용 BIM 저작 도구
- 범용 CAD 편집기
- 완전한 ERP/그룹웨어 기능
- 외부 공개 지도 서비스 수준의 대규모 소비자 트래픽 처리

## 8. 기능 요구사항

### 8.1 프로젝트 관리

- 사용자는 조직 내 프로젝트를 생성할 수 있어야 한다.
- 프로젝트에는 위치, 좌표계, 기간, 고객, 데이터 보안 등급을 설정할 수 있어야 한다.
- 프로젝트별 멤버와 권한을 관리할 수 있어야 한다.
- 프로젝트 상태는 계획, 촬영, 처리 중, 분석 중, 보고 완료, 보관으로 구분한다.

### 8.2 공간 데이터 업로드

- 사용자는 정사영상, Point Cloud, DEM, DSM, 3D Mesh, 원본 사진, GCP/CP 데이터를 업로드할 수 있어야 한다.
- 업로드된 데이터는 프로젝트와 데이터셋에 연결되어야 한다.
- 파일 업로드는 대용량 청크 업로드를 지원해야 한다.
- 업로드 실패 시 재시도할 수 있어야 한다.
- 업로드 완료 후 처리 파이프라인이 자동 또는 수동으로 실행되어야 한다.

### 8.3 3D 시각화

- 정사영상은 타일 형태로 표시되어야 한다.
- Point Cloud는 웹에서 스트리밍되어야 한다.
- 3D Mesh는 3D Tiles 형식으로 표시되어야 한다.
- 사용자는 레이어 투명도와 순서를 조정할 수 있어야 한다.
- 사용자는 거리, 면적, 높이, 체적을 측정할 수 있어야 한다.
- 분석 결과 레이어는 위험도 색상으로 표시되어야 한다.

### 8.4 GeoAI 분석 실행

- 사용자는 분석 모듈과 입력 데이터셋을 선택할 수 있어야 한다.
- 분석 실행 전 파라미터를 확인할 수 있어야 한다.
- 분석 작업은 비동기 큐에 등록되어야 한다.
- 분석 상태는 대기, 실행 중, 완료, 실패로 표시되어야 한다.
- 분석 완료 후 결과 레이어와 결과 테이블이 생성되어야 한다.
- 분석 결과는 사용자가 검수하고 승인할 수 있어야 한다.

### 8.5 보고서 생성

- 사용자는 프로젝트와 분석 결과를 선택해 보고서를 생성할 수 있어야 한다.
- 보고서 템플릿은 정부 제출용, 내부 보고용, 고객 납품용으로 구분한다.
- 보고서에는 지도 이미지, 표, 그래프, 분석 요약, 검토 의견이 포함되어야 한다.
- 보고서는 PDF로 내보낼 수 있어야 한다.
- 보고서 생성 이력과 승인 이력이 남아야 한다.

### 8.6 관리자 기능

- 시스템 관리자는 조직과 사용자를 관리할 수 있어야 한다.
- 역할별 권한을 모듈 단위로 설정할 수 있어야 한다.
- API 키를 생성, 비활성화, 회전할 수 있어야 한다.
- 인프라 상태와 작업 큐 상태를 확인할 수 있어야 한다.
- 감사 로그를 검색하고 내보낼 수 있어야 한다.
- 백업 상태와 복구 지점을 확인할 수 있어야 한다.

## 9. 비기능 요구사항

### 9.1 성능

- 일반 지도 화면 초기 로딩은 3초 이내를 목표로 한다.
- 프로젝트 목록, 데이터셋 목록, 감사 로그 등 일반 테이블 API는 1초 이내 응답을 목표로 한다.
- 대용량 3D/Point Cloud 데이터는 타일 기반 스트리밍으로 점진 로딩한다.
- GeoAI 분석은 비동기 작업으로 처리하며 UI는 작업 상태를 실시간 또는 준실시간으로 갱신한다.

### 9.2 확장성

- 조직과 프로젝트 단위로 데이터가 분리되어야 한다.
- 분석 작업자는 수평 확장 가능해야 한다.
- 타일 서버와 API 서버는 분리 배포 가능해야 한다.
- 스토리지는 온프레미스 NAS와 S3 호환 오브젝트 스토리지 모두 지원 가능한 구조로 설계한다.

### 9.3 보안

- 모든 사용자는 인증 후 접근해야 한다.
- 역할 기반 접근 제어를 적용한다.
- 프로젝트별 권한을 지원한다.
- API 키는 해시 또는 암호화된 형태로 저장한다.
- 감사 로그는 관리자도 임의 수정할 수 없도록 append-only 구조를 권장한다.
- 외부 공유 링크는 만료 시간과 접근 범위를 설정할 수 있어야 한다.

### 9.4 데이터 주권

- 고객 데이터 저장 위치를 추적할 수 있어야 한다.
- 외부 클라우드 사용 여부를 프로젝트 또는 조직 단위로 명시할 수 있어야 한다.
- 한국 서버 운영, 온프레미스 배포, 폐쇄망 배포 시나리오를 고려한다.

### 9.5 운영성

- 서버, 큐, 작업자, 스토리지, 모델 엔드포인트 상태를 모니터링해야 한다.
- 작업 실패와 오류 로그를 추적할 수 있어야 한다.
- 백업과 복구 절차가 문서화되어야 한다.
- 배포는 스테이징과 프로덕션 환경을 분리한다.

## 13. MVP 범위

MVP는 실제 고객 데모와 파일럿 구축을 목표로 한다.

포함:

- 로그인과 조직/사용자 관리
- 프로젝트 관리
- 데이터셋 업로드와 목록
- 정사영상/3D Tiles/Point Cloud 뷰어
- 기본 레이어 관리
- 2~3개 GeoAI 분석 모듈 연동
- 분석 결과 지도 표시
- 보고서 초안 생성
- 관리자 콘솔 기본 기능

제외:

- 모든 15개 GeoAI 모듈 완성
- 완전 자동 미션 비행
- 복잡한 실시간 재난 관제
- 모든 외부 시스템 연동
- 고급 과금 시스템

## 14. 릴리즈 단계

### Phase 1. MVP

목표: 데모와 파일럿 가능한 최소 제품.

주요 기능:

- 프로젝트/데이터셋/뷰어/기본 분석/보고서/관리자 콘솔

### Phase 2. Pilot

목표: 실제 고객 데이터와 운영 프로세스 검증.

주요 기능:

- 대용량 데이터 처리 안정화
- GeoAI 결과 검수
- 보고서 템플릿 고도화
- 권한과 감사 로그 강화

### Phase 3. Commercial

목표: 유료 SaaS 또는 구축형 납품 가능 수준.

주요 기능:

- 멀티테넌시 강화
- API 상품화
- 관리자 콘솔 확장
- 모델 레지스트리와 인프라 모니터링
- 백업/복구/보안 정책

### Phase 4. Enterprise

목표: 대기업/공공기관 장기 운영.

주요 기능:

- 폐쇄망 구축 옵션
- SSO
- 데이터 주권 관리
- 고급 감사 로그
- 다중 리전 또는 다중 사이트 운영
