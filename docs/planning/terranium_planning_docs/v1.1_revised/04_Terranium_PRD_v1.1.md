## 1. 제품 개요

Terranium은 드론 측량 데이터, 2D/3D 디지털 트윈, GeoAI 분석, 검수 워크플로우, 보고서 자동화를 하나의 업무 흐름으로 연결하는 B2B Spatial AI 플랫폼이다.

초기 제품은 SaaS보다 단일 고객 구축형 파일럿을 우선한다. 다만 데이터 모델, 권한 모델, 인프라 구조는 향후 SaaS와 멀티테넌트 운영으로 확장 가능하도록 설계한다.

초기 핵심 정의:

"Terranium은 산업시설·공공시설 안전 점검을 위해 드론 측량 데이터를 2D/3D로 통합 시각화하고, 균열 또는 도로 손상 분석 결과를 검수 가능한 지도 레이어와 보고서로 자동화하는 구축형 우선 Spatial AI 플랫폼이다."

## 2. 배경과 문제 정의

공공기관, 산업단지, 제조 대기업, 시설 관리 조직은 드론 촬영과 측량 결과물을 점점 더 많이 확보하고 있다. 그러나 현업에서는 다음 문제가 반복된다.

- 정사영상, LiDAR, Point Cloud, DEM/DSM, 3D Mesh, 현장 사진이 분산되어 있다.
- 드론 데이터 처리 결과를 담당자가 직접 다운로드, 변환, 캡처, 보고서화해야 한다.
- 균열, 도로 손상, 지반 변화, 침수 취약 구역 등 위험 분석이 개별 도구와 수작업에 의존한다.
- 분석 결과의 근거 데이터, 검수 이력, 승인 이력을 추적하기 어렵다.
- 내부 보고, 고객 납품, 공공 제출 자료를 매번 수작업으로 만든다.
- 데이터 접근 권한, 반출 이력, 보관 정책, 감사 로그를 통합 관리하기 어렵다.

Terranium은 이 문제를 프로젝트 기반 데이터 관리, 2D/3D Web GIS, GeoAI 결과 레이어, 검수 승인, 보고서 생성, 관리자 콘솔로 해결한다.

## 3. 제품 비전

한국의 핵심 공간 데이터를 안전하게 수집하고, 정밀하게 분석하며, 산업과 공공시설 안전 의사결정을 가속화하는 Spatial AI 운영 플랫폼.

## 4. 제품 포지셔닝

Terranium은 다음 제품군의 교차 영역에 위치한다.

- Web GIS / 3D Digital Twin Viewer
- Drone Mapping Data Platform
- GeoAI Inspection & Decision Support Platform
- Enterprise Spatial Data Governance Platform

기존 GIS, 3D 뷰어, 드론 데이터 관리 도구와 달리 Terranium은 데이터 등록, 시각화, 분석, 검수, 보고서까지 연결하는 업무형 플랫폼을 목표로 한다.

## 5. 초기 고객 도메인

### 5.1 1순위 도메인

산업시설·공공시설 안전 점검

대표 현장:

- 산업단지
- 제조 공장
- 발전소 및 에너지 시설
- 도로, 교량, 터널 주변
- 공공시설 부지
- 항만, 해양시설, 물류시설

우선 선정 이유:

- 드론 측량과 3D 디지털 트윈의 가치가 직관적이다.
- 균열, 도로 손상, 지반 변화, 침수 취약도 등 여러 분석을 "시설 안전 리스크"로 묶을 수 있다.
- 재난 예측보다 법적 책임 부담이 낮다.
- 보고서 자동화의 업무 절감 효과를 고객에게 설명하기 쉽다.

### 5.2 후순위 도메인

- 도시·국토 관리
- 재난 대응 및 침수 시뮬레이션
- 불법 건축물 탐지
- 농지·산림·탄소 분석
- 해안 침식 및 환경 모니터링

## 6. 핵심 사용자

### 6.1 시설 안전·환경 담당자

주요 니즈:

- 위험 지점과 손상 구역을 지도에서 확인
- 반복 점검 결과 비교
- 현장 작업 전 위험 구역 파악
- 내부 보고 및 외부 감사 대응
- 분석 결과의 근거 이미지와 검수 이력 확인

### 6.2 공공기관 관리자

주요 니즈:

- 시설·지역 단위 위험 현황 확인
- 분석 결과와 원본 데이터 추적
- 제출용 보고서 생성
- 권한, 감사 로그, 데이터 반출 이력 관리

### 6.3 드론 측량 운영자

주요 니즈:

- 프로젝트별 촬영 구역과 비행 정보 관리
- 센서, 고도, 중복도, 목표 GSD 기록
- GCP/CP 위치와 RTK 기준 정보 기록
- 원본 데이터 업로드와 처리 상태 확인

### 6.4 GeoAI 분석가

주요 니즈:

- 분석 대상 데이터셋 선택
- 모델 실행 또는 결과 등록
- 결과 레이어 검수
- 신뢰도, 오탐, 누락 객체 관리
- 승인된 결과를 보고서와 API로 배포

### 6.5 시스템 관리자

주요 니즈:

- 조직, 사용자, 역할, 프로젝트 권한 관리
- API 키와 사용량 관리
- 감사 로그 검색
- 데이터 주권, 반출, 보관 정책 관리
- 백업과 복구 상태 확인

## 7. 제품 목표

### 7.1 1차 목표

- 드론 측량 프로젝트를 생성하고 데이터셋을 관리할 수 있다.
- 정사영상, 3D Tiles, Point Cloud를 웹에서 시각화할 수 있다.
- 1개 대표 GeoAI 모델 결과를 지도 레이어와 테이블로 제공한다.
- 분석 결과를 사용자가 검수하고 승인할 수 있다.
- 승인된 결과를 보고서 초안으로 생성할 수 있다.
- 조직, 사용자, 프로젝트 권한, 감사 로그를 기본 관리할 수 있다.

### 7.2 2차 목표

- 추가 GeoAI 모델을 순차적으로 확장한다.
- 시간 축 기반 변화 탐지와 비교 분석을 제공한다.
- 침수 취약도 분석과 시나리오 비교 기능을 제공한다.
- 외부 시스템 연동 API를 제공한다.
- 고객별 독립 테넌트와 데이터 주권 정책을 강화한다.

### 7.3 비목표

초기 버전에서 다음은 제외한다.

- 완전 자동 드론 관제 시스템
- 실시간 드론 영상 스트리밍 관제
- 범용 BIM 저작 도구
- 범용 CAD 편집기
- 완전한 ERP/그룹웨어 기능
- SaaS 셀프 가입, 결제, 과금 시스템
- 복잡한 물리 기반 재난 시뮬레이션
- 모든 GeoAI 모델 동시 완성

### 8.6 TerraSim

TerraSim은 MVP에서 제외한다. Pilot 이후에는 "침수 취약도 분석" 또는 "시나리오 비교" 기능으로 단계적으로 검증한다.

초기 접근 원칙:

- 법적 재난 예측 시스템으로 표현하지 않는다.
- 의사결정 보조용 분석으로 명시한다.
- 결과에는 데이터 한계, 모델 한계, 검수자 승인 여부를 포함한다.

## 9. 기능 요구사항

### 9.1 프로젝트 관리

- 사용자는 조직 내 프로젝트를 생성할 수 있어야 한다.
- 프로젝트에는 위치, 좌표계, 기간, 고객, 데이터 보안 등급을 설정할 수 있어야 한다.
- 프로젝트별 멤버와 권한을 관리할 수 있어야 한다.
- 프로젝트 상태는 계획, 촬영, 처리 중, 분석 중, 검수 중, 보고 완료, 보관으로 구분한다.

### 9.2 공간 데이터 업로드

- 사용자는 정사영상, Point Cloud, DEM/DSM, 3D Mesh, 원본 사진, GCP/CP 데이터를 업로드할 수 있어야 한다.
- 업로드는 대용량 청크 업로드를 지원해야 한다.
- 업로드 중단 후 재개할 수 있어야 한다.
- 업로드 완료 후 무결성 검증을 수행해야 한다.
- 업로드 완료 후 처리 파이프라인이 자동 또는 수동으로 실행되어야 한다.

### 9.3 3D 시각화

- 정사영상은 COG 또는 raster tile 형태로 표시되어야 한다.
- 3D Mesh는 3D Tiles 형식으로 표시되어야 한다.
- Point Cloud는 COPC 또는 3D Tiles point cloud 형태로 표시되어야 한다.
- 분석 결과 레이어는 위험도 또는 상태 색상으로 표시되어야 한다.
- 사용자는 레이어 투명도, 순서, 표시 여부를 조정할 수 있어야 한다.

### 9.4 GeoAI 분석 실행과 검수

- 사용자는 분석 모듈과 입력 데이터셋을 선택할 수 있어야 한다.
- 분석 실행 전 파라미터와 모델 버전을 확인할 수 있어야 한다.
- 분석 작업은 비동기 큐에 등록되어야 한다.
- 분석 상태는 대기, 실행 중, 완료, 실패, 취소로 표시되어야 한다.
- 분석 완료 후 결과 레이어와 결과 테이블이 생성되어야 한다.
- 분석 결과는 draft, reviewed, approved, rejected 상태를 가져야 한다.
- 검수자는 객체 삭제, 등급 변경, 코멘트 입력을 할 수 있어야 한다.

### 9.5 보고서 생성

- 사용자는 승인된 분석 결과를 기반으로 보고서를 생성할 수 있어야 한다.
- 보고서에는 지도 이미지, 분석 결과 표, 요약, 검토 의견, 책임 고지 문구가 포함되어야 한다.
- 보고서는 PDF로 내보낼 수 있어야 한다.
- 보고서 생성 이력과 승인 이력이 남아야 한다.

### 9.6 데이터 반출과 공유

- 외부 공유 링크는 만료 시간과 접근 범위를 설정할 수 있어야 한다.
- 원본 데이터 다운로드와 보고서 반출은 감사 로그에 기록되어야 한다.
- 보안 등급이 높은 프로젝트는 다운로드 승인 절차를 적용할 수 있어야 한다.

## 10. 비기능 요구사항

### 10.1 성능

API 목표:

- 일반 테이블 API p95 응답 1초 이내
- 프로젝트 상세 API p95 응답 1.5초 이내
- 작업 상태 갱신 지연 5초 이내

지도 목표:

- 일반 지도 화면 초기 로딩 3초 이내
- 첫 타일 표시 2초 이내
- 레이어 ON/OFF 반응 500ms 이내

데이터 규모 기준:

- 정사영상 5GB: MVP 기준 처리 가능
- 정사영상 20GB: MVP-B 기준 처리 검증
- 정사영상 100GB: Pilot 이후 성능 검증
- Point Cloud 1억 점: MVP-B 기준 표시 검증
- Point Cloud 5억 점 이상: Pilot 이후 최적화

### 10.2 확장성

- 조직과 프로젝트 단위로 데이터가 분리되어야 한다.
- 분석 작업자는 수평 확장 가능해야 한다.
- 타일 서버와 API 서버는 분리 배포 가능해야 한다.
- 스토리지는 S3 호환 오브젝트 스토리지와 온프레미스 NAS를 모두 고려한다.

### 10.3 보안

- 모든 사용자는 인증 후 접근해야 한다.
- 역할 기반 접근 제어와 프로젝트 ACL을 적용한다.
- API 키는 해시 저장하고 secret은 최초 1회만 표시한다.
- 감사 로그는 append-only 구조를 기본 원칙으로 한다.
- presigned URL은 짧은 만료 시간을 가져야 한다.
- 파일 업로드 검증과 악성코드 검사 옵션을 고려한다.

### 10.4 데이터 주권

- 고객 데이터 저장 위치를 추적할 수 있어야 한다.
- 외부 클라우드 사용 여부를 프로젝트 또는 조직 단위로 명시할 수 있어야 한다.
- 한국 서버 운영, 온프레미스 배포, 폐쇄망 배포 시나리오를 고려한다.
- 고객 데이터 폐기와 보관 정책을 프로젝트 단위로 설정할 수 있어야 한다.

### 10.5 법적 책임과 승인

- GeoAI와 TerraSim 계열 결과는 의사결정 보조 자료로 명시한다.
- 보고서에는 자동 분석 결과의 한계와 최종 검수자 승인 여부를 포함한다.
- 재난, 안전, 구조 진단의 최종 판단은 전문 검수자 또는 고객 책임자의 승인 절차를 거쳐야 한다.
- 모델 정확도, 입력 데이터 품질, 촬영 조건에 따른 한계를 문서화한다.
