## 1. 총평

현재 PRD와 구현 계획은 Terranium을 단순 3D 뷰어가 아니라 드론 데이터 수집, 공간 데이터 처리, 3D 디지털 트윈, GeoAI 분석, 보고서, 관리자 콘솔까지 연결하는 엔드투엔드 플랫폼으로 정의하고 있다. 방향성은 타당하다.

다만 현재 문서는 "최종 제품 청사진"에 가깝고, "첫 개발 착수용 실행 계획"으로는 범위가 과대하다. 특히 MVP에 3D Tiles, Point Cloud, 2~3개 GeoAI, 보고서, 관리자 콘솔, 감사 로그, API 키까지 동시에 넣으면 16주 일정은 매우 공격적이다.

핵심 조정 방향은 다음과 같다.

- 초기 고객 도메인을 하나로 고정한다.
- MVP는 데이터 업로드, 정사영상/3D 뷰어, 단일 GeoAI 분석, 보고서 초안까지로 축소한다.
- TerraSim은 MVP에서 제외하고 Pilot 이후 실험 기능으로 둔다.
- SaaS와 온프레미스를 동시에 완성하려 하지 말고, 구축형 파일럿 우선 구조로 설계한다.
- 법적 책임이 있는 "예측" 표현은 "의사결정 보조"와 "분석 참고"로 낮춘다.

## 2. 맞는 판단

### 2.1 제품 포지셔닝

Web GIS, 드론 데이터 플랫폼, GeoAI Decision Intelligence의 교차 영역으로 정의한 것은 맞다. 디오아이의 드론 측량, 데이터 처리, 공공·산업 고객 접점과도 연결된다.

### 2.2 기술 스택

Next.js, React, TypeScript, FastAPI, PostgreSQL/PostGIS, Redis, 비동기 큐, GDAL, PDAL, COG, COPC, 3D Tiles 방향은 적절하다.

검토 기준:

- CesiumJS는 3D Tiles, 지형, point cloud 계열 표시와 잘 맞는다.
- MapLibre GL JS는 2D 지도, raster tile, vector tile 화면에 적합하다.
- COG는 대용량 GeoTIFF를 HTTP 기반 부분 읽기와 타일 제공 구조로 운영하기 좋다.
- COPC는 대용량 point cloud의 클라우드 친화 저장 포맷으로 적절하다.

### 2.3 모듈 구성

Dashboard, Viewer, Mission/QC, Data Management, GeoAI, Report, Admin Console의 분리는 자연스럽다. 실제 업무 흐름도 프로젝트 생성, 촬영, 업로드, 처리, 분석, 검수, 보고서 순서로 잘 잡혀 있다.

## 3. 틀렸거나 위험한 부분

### 3.1 MVP 범위가 너무 넓다

현재 MVP에는 로그인, 조직/사용자, 프로젝트, 업로드, 정사영상, 3D Tiles, Point Cloud, 기본 레이어, 2~3개 GeoAI, 보고서, 관리자 콘솔이 모두 포함되어 있다.

이 범위는 "데모 가능한 제품"이 아니라 "파일럿 운영형 제품"에 가깝다. 특히 Point Cloud 처리, 3D Tiles 변환, GeoAI 2~3개, 보고서 자동화는 각각 별도 리스크가 큰 작업이다.

권장 수정:

- MVP-A: 고객 데모용 8주
- MVP-B: 파일럿용 16주
- Pilot: 실제 고객 데이터 운영 12주

### 3.2 초기 고객 도메인이 고정되지 않았다

공공안전, 산업시설, 국토관리, 도로, 불법 건축물, 침수, 균열을 모두 열어두면 데이터셋, 모델, 보고서, 영업 메시지가 분산된다.

권장 1순위:

- 산업시설·공공시설 안전 점검

이유:

- 드론 측량과 3D 디지털 트윈의 시각적 설득력이 높다.
- 균열, 도로 손상, 침수 위험, 지반침하를 하나의 "시설 안전 리스크" 범주로 묶을 수 있다.
- 정부 재난 예측보다 법적 책임 부담이 낮다.

### 3.3 GeoAI 2~3개 동시 개발은 위험하다

초기 GeoAI 모듈은 모델 정확도, 학습 데이터 확보, 라벨링 품질, 현장 검수 절차가 모두 필요하다. 문서에는 모델 후보가 많지만 각 모델별 입력 데이터, 정답 데이터, 성능 기준, 실패 처리 기준이 없다.

권장 수정:

- MVP는 "분석 모듈 프레임워크 + 1개 대표 모델"만 구현한다.
- 나머지는 데모용 mocked result 또는 rule-based baseline으로 둔다.

추천 1순위 모델:

- 도로 손상 또는 균열 탐지

이유:

- 이미지 기반 객체 탐지/세그멘테이션으로 범위가 명확하다.
- 결과를 지도 레이어와 보고서로 보여주기 쉽다.
- 침수 예측보다 법적 책임과 물리 모델 리스크가 낮다.

### 3.4 TerraSim은 MVP에 넣으면 안 된다

침수 시뮬레이션은 DEM/DSM, 배수 조건, 강우량, 유역, 하수관망, 수문 모델이 얽힌다. 단순 위험도 래스터로 시작할 수는 있지만, "시뮬레이션"이라는 이름은 고객에게 과도한 정확도 기대를 만든다.

권장 수정:

- MVP에서는 TerraSim 제외
- Pilot에서는 "침수 취약도 분석"으로 명명
- Commercial 이후에 "시나리오 비교"로 확장

### 3.5 SaaS와 온프레미스 동시 지원은 과하다

PRD는 B2B SaaS로 시작하지만, 데이터 주권과 폐쇄망, 온프레미스도 함께 고려한다. 공공·대기업 대상이면 맞는 방향이지만, 초기 개발에서는 운영 모델이 갈라진다.

권장 수정:

- 제품 구조는 멀티테넌시를 고려한다.
- 실제 배포 우선순위는 "단일 고객 구축형 파일럿"으로 둔다.
- SaaS 과금, 셀프 가입, 공개 결제, 대규모 멀티테넌트 운영은 후순위로 둔다.

### 3.6 성능 목표가 구체적이지 않다

"초기 로딩 3초"는 좋은 목표지만 데이터 크기 기준이 없다. 3D Tiles, Point Cloud, 정사영상은 데이터 규모에 따라 성능이 크게 달라진다.

추가해야 할 기준:

- 정사영상: 5GB, 20GB, 100GB 처리 기준
- Point Cloud: 1억 점, 5억 점, 10억 점 기준
- 3D Tiles: 타일셋 총량, 평균 타일 크기, LOD 기준
- 업로드: 단일 파일 최대 크기, 청크 크기, 재개 가능 여부
- 처리 시간: 데이터 크기별 목표 시간

### 3.7 데이터 모델이 운영 필수 항목을 일부 빠뜨렸다

추가 권장 엔티티:

- UploadSession
- UploadPart
- DatasetAsset
- TileAsset
- LayerStyle
- ProjectMember
- AccessPolicy
- ReportTemplate
- ReportApproval
- ModelVersion
- ModelDeployment
- JobEvent
- Notification
- DataRetentionPolicy
- ExternalShare
- WebhookEndpoint

### 3.8 보안 계획은 방향은 맞지만 구현 기준이 부족하다

RBAC, 감사 로그, API 키, 백업 암호화는 적절하다. 다만 공공·대기업 제안에 쓰려면 다음이 추가되어야 한다.

- 개인정보 포함 가능성 분류
- 파일 업로드 악성코드 검사
- 객체 스토리지 presigned URL 정책
- 다운로드/반출 승인 워크플로우
- 관리자 권한 오남용 방지
- 감사 로그 WORM 또는 별도 저장소 보관
- SSO/SAML/OIDC 우선순위
- 망분리/폐쇄망 업데이트 절차

## 4. 권장 MVP 재정의

### 4.1 MVP-A: 8주 고객 데모

목표:

- 영업, 투자, 고객 미팅에서 Terranium의 핵심 가치를 보여주는 데모

포함:

- 로그인 화면
- 대시보드
- 프로젝트 목록과 상세
- 사전 처리된 정사영상 표시
- 사전 처리된 3D Tiles 표시
- 사전 처리된 Point Cloud 또는 샘플 3D Tiles point cloud 표시
- 레이어 패널
- 측정 도구 기본
- 1개 GeoAI 분석 결과 표시
- 분석 결과 테이블
- 보고서 PDF 샘플 생성

제외:

- 실제 대용량 업로드 처리
- 모델 학습/실시간 추론
- 복잡한 권한 매트릭스
- TerraSim
- 백업/복구 UI
- API 키 과금/사용량

### 4.2 MVP-B: 16주 파일럿

목표:

- 실제 고객 데이터 1~2건을 받아 업로드, 처리, 시각화, 분석, 보고서까지 운영 검증

포함:

- 조직/사용자/프로젝트 권한
- 청크 업로드
- COG 생성
- 3D Tiles 등록
- Point Cloud 변환 1종 표준화
- 처리 작업 큐
- 분석 작업 큐
- 1개 GeoAI 모델 실제 실행
- 검수 상태
- 보고서 템플릿 1~2종
- 감사 로그
- 관리자 기본 화면

제외:

- 다중 모델 운영 자동화
- 완전한 모델 레지스트리
- 복잡한 시뮬레이션
- SaaS 과금
- 완전한 폐쇄망 배포 자동화

## 5. 우선 의사결정

개발 착수 전에 반드시 결정해야 한다.

1. 초기 고객 도메인
2. 첫 번째 GeoAI 모델
3. 첫 번째 데이터 포맷 조합
4. 배포 모델
5. 법적 책임 문구와 승인 절차
6. 파일럿 성공 기준

권장 결정안:

- 초기 고객 도메인: 산업시설·공공시설 안전 점검
- 첫 번째 GeoAI 모델: 균열 또는 도로 손상 탐지
- 첫 번째 데이터 포맷: COG + 3D Tiles + LAZ/COPC
- 배포 모델: 단일 고객 구축형 파일럿 우선
- 법적 책임: 의사결정 보조, 최종 판단은 전문 검수자 승인
- 성공 기준: 업로드부터 보고서 초안까지 기존 대비 50% 이상 시간 단축

## 6. 수정이 필요한 문서 항목

### PRD 수정

- 제품명을 "B2B SaaS"로만 정의하지 말고 "SaaS/구축형 모두 가능한 B2B 플랫폼"으로 수정
- MVP 범위를 MVP-A와 MVP-B로 분리
- TerraSim을 2차 또는 3차 목표로 이동
- 초기 GeoAI 2~3개를 1개 대표 모델로 축소
- 성공 지표에 파일럿 기준 수치를 추가
- 법적 책임과 검수 승인 요구사항을 별도 섹션으로 추가

### 구현 계획 수정

- 16주 계획을 "파일럿 MVP"로 명확히 변경
- 8주 데모 트랙을 별도 추가
- 데이터 규모별 성능 기준 추가
- UploadSession, ProjectMember, ReportTemplate 등 누락 엔티티 추가
- GeoAI 모델 인터페이스에 평가 기준, 실패 기준, 검수 기준 추가
- 관리자 콘솔은 MVP에서 기본 기능만 남기고 고급 인프라/백업 UI는 Pilot 이후로 이동

## 7. 권장 개발 순서

1. 샘플 데이터셋 확보
2. 정사영상 COG 변환과 2D 지도 표시 PoC
3. 3D Tiles와 Point Cloud 표시 PoC
4. 프로젝트/데이터셋/레이어 메타데이터 모델 확정
5. 업로드와 처리 작업 큐 구현
6. GeoAI 결과 레이어 스키마 확정
7. 1개 모델 또는 mock inference 연동
8. 검수 UI 구현
9. 보고서 템플릿 구현
10. 감사 로그와 권한 강화

## 8. 결론

현재 계획은 방향은 맞지만, 개발 착수 전 범위 축소와 우선순위 고정이 필요하다. Terranium의 첫 성공은 모든 기능을 한 번에 만드는 것이 아니라, 특정 고객 문제 하나를 데이터 수집부터 보고서까지 끊기지 않게 해결하는 것이다.

추천하는 첫 제품 정의는 다음과 같다.

"Terranium은 산업시설·공공시설 안전 점검을 위해 드론 측량 데이터를 2D/3D로 통합 시각화하고, 균열 또는 도로 손상 분석 결과를 검수 가능한 지도 레이어와 보고서로 자동화하는 구축형 우선 Spatial AI 플랫폼이다."
