# AGENTS.md

이 문서는 Terranium 프로젝트에서 AI 에이전트가 협업할 때 따르는 기본 역할과 운영 원칙을 정의한다.

현재는 초기 기준만 정리한 상태이며, 구체적인 작업 매뉴얼은 Codex와 사용자 간 협업 과정에서 점진적으로 보완한다.

## 현재 작업공간 범위

이 작업공간은 Terranium 전체 원본 프로젝트가 아니라 **프론트엔드만 분리해서 개발하기 위한 별도 작업 디렉터리**다.

- 이 작업공간의 목표는 Terranium 프론트엔드 화면, 프론트엔드 상태, mock API, viewer PoC, UI/UX 구현물을 만든 뒤 나중에 필요한 프론트엔드 파일만 원본 프로젝트에 옮기는 것이다.
- 백엔드 개발, FastAPI 서버 구현, DB 스키마 작성, 인프라 구성은 이 작업공간의 기본 범위가 아니다.
- 문서에는 향후 전체 Terranium 플랫폼 기준의 백엔드/API/인프라 계획이 포함될 수 있으나, 이 작업공간에서 실제 구현할 대상은 프론트엔드에 한정한다.
- 사용자가 명시적으로 요청하지 않는 한 `apps/api/`, DB, Docker/Kubernetes, 운영 인프라 파일을 새로 구현하지 않는다.
- 이 작업공간에서는 사용자가 의도적으로 git을 제거했으므로 git 기반 브랜치, 커밋, push, merge 작업을 수행하지 않는다.
- 에이전트는 `git status`, `git diff`, `git log`, 브랜치 생성, 커밋, push, merge를 전제로 작업 계획을 세우지 않는다.
- 변경 이력과 진행 상태는 파일 변경 내용, 문서, `docs/reports/`의 한국어 작업 보고서를 기준으로 관리한다.

## 참여 에이전트

현재 `terranium_front` 작업공간에서는 기본적으로 **Codex가 메인 프론트엔드 개발을 담당**한다.

- Codex
- Claude

Claude는 메인 개발이나 코드 수정을 담당하지 않는다. 단, 사용자가 지시한 경우 **브라우저 MCP 기반 검증만 담당**한다.

## 기본 역할

아래 역할은 Terranium 전체 원본 프로젝트 기준의 일반 원칙과, 현재 `terranium_front` 프론트엔드 분리 작업공간의 실제 운영 방식을 함께 정의한다.

현재 `terranium_front` 작업공간에서는 **Codex가 메인 프론트엔드 개발자**다. Claude는 브라우저 MCP 검증 보조 역할만 수행한다. 두 에이전트 모두 백엔드가 아니라 프론트엔드 범위만 다룬다.

### Codex

Codex는 프로젝트의 설계, 검증, 테스트, 개선점 도출, 업무 분해, 작업 지시를 담당한다.

주요 책임:

- 제품/기술 설계 방향 정리
- 제품 구현 과정의 기본 의사결정
- 저장소 구조와 작업 기준 검토
- 테스트 및 검증 전략 수립
- 구현 결과 리뷰
- 개선점과 리스크 도출
- 사용자와 협의해 `AGENTS.md` 및 관련 작업 기준 보완

Codex는 사용자가 직접 결정하라고 명시한 사안을 제외하고, 구현 방향과 기술 선택에 대한 기본 결정을 내린다.

예를 들어 다음과 같은 사안은 Codex가 판단한다.

- 첫 대표 GeoAI 모델 선택
- MVP-A viewer 구현 방식 선택
- mock API 방식 선택
- 구현 우선순위
- 테스트와 검증 기준

사용자는 필요할 경우 Codex의 결정을 수정하거나 직접 지시할 수 있다. 단, 사용자가 별도로 수정하지 않는 한 Codex의 결정을 기본 기준으로 따른다.

Terranium 전체 원본 프로젝트 기준에서는 Codex가 기본적으로 실제 코딩 작업을 수행하지 않는다.

단, 사용자가 Codex에게 직접 수정을 지시한 경우에는 Codex가 파일 수정, 코드 작성, 구조 정리, 문서 작성 등을 수행할 수 있다.

현재 `terranium_front` 작업공간에서 Codex의 역할:

- Codex는 메인 프론트엔드 개발자다.
- 레퍼런스 디자인 사진이 제공되면 전체 화면과 사용자 경험을 최대한 원본 디자인과 똑같이 맞추는 것을 주 작업으로 한다.
- 페이지 구조, 레이아웃, 공통 컴포넌트, 스타일, 반응형 화면, 상태 흐름, mock data 연동 등 프론트엔드 구현 전반을 주도한다.
- 구현 중 레퍼런스 디자인과 차이가 나는 부분, 애매한 인터랙션, 화면 구성 판단이 필요한 부분은 사용자에게 확인한다.
- 설계, 구현, 검증, 테스트, 문서 정리를 한 에이전트 흐름 안에서 수행한다.
- 백엔드, DB, 인프라 구현은 담당하지 않는다.

### Claude

Claude는 현재 `terranium_front` 작업공간에서 **브라우저 MCP 검증만 담당**한다.

Claude의 역할:

- Codex가 구현한 화면을 브라우저 MCP로 열어 실제 렌더링을 확인한다.
- 레퍼런스 디자인 이미지와 현재 화면을 비교해 차이를 한국어로 보고한다.
- viewport, URL, 콘솔 에러, 레이아웃 깨짐, 색상/간격/정렬 차이를 확인한다.
- Playwright MCP 스크린샷을 기준으로 싱크로율 개선 포인트를 제안한다.
- 코드 수정, 파일 생성, dependency 설치, git 작업은 하지 않는다.
- Codex가 요청한 검증 범위 밖의 구현 판단을 임의로 넓히지 않는다.

## 협업 원칙

- 현재 `terranium_front` 작업공간에서는 Codex가 프론트엔드 메인 구현을 맡는다.
- Claude는 브라우저 MCP 검증만 맡는다.
- Codex는 레퍼런스 디자인 사진을 기준으로 프론트엔드 화면을 최대한 동일하게 맞추는 것을 우선한다.
- Codex는 전체 화면과 제품 경험 구현, 특정 부분의 정밀 보정, 검증, 문서 정리를 모두 담당한다.
- Claude가 MCP 검증에서 발견한 차이는 Codex가 판단해 구현에 반영한다.
- 요구사항이 불명확하거나 기존 구조와 충돌하는 경우 사용자의 확인을 받는다.
- 세부 매뉴얼이 아직 없으므로, 에이전트는 임의의 대규모 구조 변경이나 기술스택 변경을 피한다.
- 기존 문서와 저장소 구조를 우선 확인한 뒤 작업한다.
- 사용자의 명시적 지시가 기존 기준과 충돌하면, Codex는 기준 변경 여부를 판단하고 필요한 경우 사용자에게 확인한다.
- 새로 합의된 규칙은 가능하면 `AGENTS.md` 또는 관련 문서에 반영한다.

## Codex 직접 작업 기준

현재 `terranium_front` 작업공간에서는 Claude에게 전달할 작업 지시를 기본으로 작성하지 않는다. Codex가 사용자와 직접 협의하고, 필요한 경우 바로 프론트엔드 파일을 구현·수정·검증한다.

다만 큰 작업을 시작할 때는 사용자가 중간에 범위와 완료 기준을 판단할 수 있도록 아래 형식으로 작업 범위를 정리한 뒤 진행한다.

사용자가 나중에 Claude를 다시 투입하라고 명시하면, 그 시점에 별도 Claude 전달용 티켓 형식을 다시 정리한다.

### 언어 기준

- 사용자에게 설명하는 내용은 한국어로 작성한다.
- 작업 범위, 결정 사항, 검증 기준은 한국어로 작성한다.
- 파일명, 브랜치명, 타입명, 명령어, API 경로, 코드 식별자는 원문을 유지한다.
- 영어로 작성하는 것은 `docs/mvp/` 같은 세부 구현 계획 문서에 한정한다.

### 작성 원칙

- 막연한 요청이 아니라 실행 가능한 단위로 쪼갠다.
- 현재 `terranium_front` 작업공간에서는 브랜치명 대신 작업공간 범위, 수정 범위, 완료 기준을 반드시 포함한다.
- 원본 프로젝트 또는 git 저장소에서 작업하는 경우에만 브랜치명을 포함한다.
- `Required Work`는 번호 매김으로 적고, 각 항목 안에 필요한 파일/타입/데이터/엔드포인트를 sub-bullet로 구체화한다.
- 큰 구현 작업에서는 한국어 완료 보고서를 작성한다. 단순 수정이나 사용자와 실시간으로 진행하는 작은 작업에서는 최종 답변으로 대체할 수 있다.
- 검증 명령과 보고서 작성 기준을 분리해 함께 적는다.
- out-of-scope와 금지 사항을 `Goal` / `Constraints`에 명확히 쓴다.
- 사용자가 중간에 읽고 판단할 수 있도록 과도한 추상화 없이 구체적으로 쓴다.

## 문서 및 보고 규칙

- 세부 구현 계획서는 에이전트 작업 편의를 위해 영어로 작성한다.
- `docs/reports/`의 한국어 작업 보고서는 계획 수립 단계가 아니라 실제 세부 구현 또는 검증 작업이 완료된 뒤 작성한다.
- 현재 `terranium_front` 작업공간에서는 Codex가 실제 구현을 수행하므로, 필요한 경우 Codex가 한국어 작업 보고서를 작성한다.
- 단순 수정이나 짧은 확인 작업은 별도 보고서 대신 사용자에게 최종 답변으로 결과를 보고할 수 있다.
- 보고서 파일명은 `YYYY-MM-DD_<short-topic>_작업보고.md` 형식을 권장한다.
- 보고서에는 작업 목적, 작업 범위, 구현/검증 내용, 결과, 계획 대비 완료 여부 체크리스트, 남은 이슈, 다음 작업 제안을 포함한다.
- 사용자가 프로젝트 진척도를 확인하는 기준은 `docs/reports/`의 한국어 보고서다.

## Git 및 브랜치 운영

현재 `terranium_front` 작업공간에는 git을 사용하지 않는다.

- 사용자가 의도적으로 git을 제거했으므로 이 작업공간에서는 브랜치 전략을 적용하지 않는다.
- 에이전트는 git branch 생성, checkout, commit, push, merge를 수행하지 않는다.
- 에이전트는 git 저장소 존재를 전제로 검증하거나 보고하지 않는다.
- 작업 결과는 파일 시스템의 변경 내용과 문서 업데이트로만 관리한다.
- 나중에 사용자가 필요한 프론트엔드 파일만 원본 Terranium 프로젝트에 옮긴다.
- 원본 프로젝트에 옮긴 뒤의 git 운영은 원본 프로젝트의 별도 기준을 따른다.
- 이 문서의 과거 `develop`, `feature/*`, `main` 브랜치 규칙은 현재 분리 프론트엔드 작업공간에는 적용하지 않는다.

## 현재 주의사항

- 세션 재시작 후 현재 프론트 구현 상태와 이어서 할 작업은 `docs/CODEx_HANDOFF.md`를 먼저 확인한다.
- 현재 프로덕션 랜딩 사이트는 `apps/landing/`에 있다.
- 현재 분리 작업공간의 실제 개발 대상은 프론트엔드이며, 제품 앱은 `apps/web/`에 둔다.
- `apps/api/`와 백엔드 구현 계획 문서는 원본 Terranium 플랫폼 기준 참고사항이며, 이 작업공간에서 백엔드를 구현하지 않는다.
- 저장소 구조 기준은 `docs/architecture/REPOSITORY_STRUCTURE.md`를 따른다.
- 문서 기준과 버전 관리는 `docs/README.md`를 따른다.
- 프론트엔드 외 범위의 프레임워크 보일러플레이트를 임의로 생성하지 않는다.
